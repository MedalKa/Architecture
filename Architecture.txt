Я решил рассматривать клиент-серерную многопользовательскую систему на примере приложения "Госуслуги".
Для примера прикрепляю Sequence UML диаграмму.
У данного приложения трёхуровневая архитектура.
У пользователя имеется клиент. В данном случаем будет "тонкий" клиент, т.к. у пользователя будет только графический интерфейс, 
все операции и логика будут находиться на сервере.
Сервером будет сервер Java-приложений. Будет возможность использовать данный сервер для различных платформ, останется только менять оболочку. 
Через сервер будет осуществляться доступ к СУБД. Он обеспечит защиту и безопасность данных, у пользователя не будет прямого доступа к данным.
в БД будет находиться вся информация о пользователях(паспортные данные, ИНН, штрафы, налоги и т.д.)
Между собой компоненты взаимодействуют через промежуточное ПО через запросы(примеры в UML диаграмме). К БД доступ получаем через универсальное ПО доступа.
Т.е. пользователь делает запрос, он отправляется на сервер, сервер делает запрос из БД, получает ответ, обрабатывает и отправляет пользователю ответ.
Производительность: данная архитектура не зависит от ресурсов клиента, зависит только от его скорости интернета, пользователю нужно только отправить запрос
и получить ответ, все операции выполнятся на сервере быстро.
Надежность:есть возможность выхода из строя всей системы, если сломается сервер, поэтому должны быть резервные сервера, что обеспечит практически бесперебойную работу.
Безопасность: в целом, данная система безопасна. У пользователя нет прямого доступа к БД. А при запросах мы можем делать проверки доступа.
